# Sage & Spoon 🌿

**Live app: [poetac.github.io/sage-and-spoon](https://poetac.github.io/sage-and-spoon/)**

A calm weekly meal planner for a household where the chef cooks for someone with **gestational diabetes (GD)**. Every meal — built-in or AI-generated — follows GD rules: low-GI carbs paired with protein/fat, per-meal carb caps, no added sugars, no juice, no white rice/bread.

> **Not medical advice.** Carb targets default to common GD guidance (≤30g breakfast, ≤45g lunch/dinner, ≤20g snacks) but are fully editable — follow your dietitian's numbers, always.

## What it does

Six tabs, all working fully offline once installed (AI features need a connection):

- **Plan** — a real-date Mon–Sun grid with 6 slots/day (3 meals + AM/PM/bedtime snacks). Drag-and-drop on desktop, tap-to-move on touch. Swap any meal from the built-in cookbook, or ask Claude for a fresh idea. Undo any edit; browse and restore past weeks from history.
- **Cookbook** — browse the full **~630-recipe library** with search, facet filters, and sort. Every recipe has a photo (a browseable gallery where available) and you can add your own; heart favorites for a one-tap "My Saved Recipes" view, or hide recipes you'll never cook.
- **Ingredients** — list what's in the kitchen; get GD-compliant meal suggestions ranked by ingredient overlap, with matched/missing chips and one-tap add-to-plan.
- **Shopping List** — auto-built from the week, grouped by aisle, scaled to your servings. Remove auto items or add your own (edits persist for the week), then print, copy, share (iOS share sheet), or download.
- **Log** — the standard gestational-diabetes blood-sugar checks (fasting + post-meal, 1h or 2h per your care team's plan), tracked by date with in-range/high/low status shown as **text + colour** (never colour alone). Trend sparklines, a weekly summary, and CSV export for appointments. Once you've logged a meal a few times, **Meal patterns** shows the average reading that followed it — explicitly framed as something to discuss with your care team, never as a diagnosis. *Not medical advice; targets are editable to match your care team's numbers.*
- **Settings** — preferences quiz, carb targets, blood-sugar targets, servings, API key, backup/restore, full reset.

First run shows a step-by-step preference quiz (cuisines, proteins, vegetables, dislikes, allergies, textures, spice, cook time), then generates a starter week instantly. Allergies are hard-excluded everywhere.

It's an installable PWA — add it to your phone's home screen and the plan, cookbook (~630 recipes, photos included), shopping list, and blood-sugar log keep working fully offline (the AI features still need a connection). All data — preferences, plan, log — stays on-device in `localStorage`/IndexedDB; there's no backend, no account, no server ever seeing your data.

## Two modes

| | No API key | With a Claude API key |
|---|---|---|
| Weekly plan | Built from the ~630-recipe cookbook | "Generate Full Week" personalizes with Claude |
| Cookbook growth | Fixed (but large) library | "Grow cookbook" adds vetted AI meals permanently |
| Swaps | Cookbook swap | Cookbook swap **+** ✨ AI swap |
| Ingredient ideas | Cookbook matching | Tailored suggestions from Claude |

Everything works fully with no key. Add one in Settings to unlock the AI features.

**Security note:** the key is stored in `localStorage` and calls the Anthropic API directly from the browser (`anthropic-dangerous-direct-browser-access`). That's acceptable for personal, self-hosted use on your own devices — for anything more, route calls through a small backend proxy instead. Never commit a key to this repo.

## Quick start

```bash
npm install
npm run dev
```

Requires Node 20+. Tailwind v4 is compiled at build time via the `@tailwindcss/vite` plugin and imported in `src/styles.css` (`@import "tailwindcss"`) — no CDN script, no separate config file.

## Project shape

```
src/
  App.jsx                  state, persistence, and composition
  styles.css               design tokens + component classes (Tailwind covers layout)
  data/meals.js            77 hand-written CORE_MEALS, quiz options, allergen/dislike maps,
                           default settings (carb caps + blood-sugar targets)
  data/generated-meals.js  ~550 curated recipes (auto-generated by the pipeline)
  data/recipe-images.js    per-recipe photo galleries (self-hosted WebP + CC attribution)
  lib/                     pure logic: storage, dates, utils, planner, shopping, claude,
                           nutrition, image, userPhotos, pwa, glucose (blood-sugar log)
  components/              primitives, NutritionPills, RecipeImage, MealCard, MealDetail,
                           Onboarding, PrefsFields, WeekHistory, ErrorBoundary, Sparkline,
                           GlucoseTab, and the six tabs (Plan, Cookbook, Ingredients,
                           Shopping, Log, Settings)
scripts/                   recipe-library pipeline (report → generate → curate → promote)
```

`MEAL_DB = [...CORE_MEALS, ...GENERATED_MEALS]`. State persists to `localStorage` (`ss_*` keys, deep-merged with defaults on load so new settings/features reach existing installs) with an in-memory fallback for sandboxed contexts; cook-supplied recipe photos live in IndexedDB. Backup/restore/reset covers every data type — plan, cookbook edits, prefs, and the blood-sugar log — in one JSON file.

## Blood-sugar log

The **Log** tab tracks the standard gestational-diabetes checks: fasting on waking, plus one post-meal reading per main meal (1 hour ≤140 mg/dL or 2 hours ≤120 mg/dL — your care team's regimen, picked in Settings). Status is always shown as text *and* colour together, never colour alone. A weekly summary and per-slot trend sparklines give an at-a-glance read on the week; **CSV export** hands the data to a care team in appointment-ready form.

Once a meal has been logged with a few readings, **Meal patterns** (also in the Log tab) shows the average reading that followed it — deliberately framed as a pattern to discuss with your care team, not a cause-and-effect claim, and gated behind a minimum sample size so a single logged day never reads as a trend.

**Not medical advice.** Targets are pre-filled with common GD guidance but are fully editable in Settings — always follow your care team's numbers.

## Growing the cookbook

The cookbook is grown offline with a gap-driven, quality-gated pipeline in `scripts/` — measure coverage, generate recipes that fill the gaps, curate by hand, then promote into the bundle:

```bash
npm run recipes:report     # coverage vs targets in scripts/lib/config.mjs
npm run recipes:generate   # AI drafts for the largest gaps (needs ANTHROPIC_API_KEY)
npm run recipes:promote    # validate curated recipes into src/data/generated-meals.js
npm run images:fetch       # resolve openly-licensed photos for recipes missing one
npm run images:self-host   # download fetched photos → local optimised WebP (offline)
```

Recipes can also be hand-authored (no API key needed) and pre-checked with `node scripts/validate-drafts.mjs <file.json>` before promoting — it runs a draft through the app's own nutrition estimator and vetting rules (ingredient recognition, computed macros, GD carb-pairing, carb-calibration drift) so problems surface before promotion instead of after. Every recipe passes the same safety gates as the runtime path (carb caps, allergen/dislike exclusions, categories, ≤8 ingredients, required steps). See `scripts/README.md` for the full workflow.
